開発の目的

フーリエ変換（fourier transformation）を作成し、それを使った信号処理アプリケーション（応用プログラム）を開発する。MATLABやMathematicaなどを援用してはならない。必ず所定のプログラミング言語を用いて自分たちの手で作り出し、それをベースにしてアプリケーションに仕立て上げること。

この開発の目的は、実際にフーリエ変換を一から作り始めて、オブジェクト指向プログラムとしてライブラリ化し、それを応用するまでの一連の作業（工程）を演習することにより、信号処理とは何かを会得すると共に、かかる開発要員が品質の高いソフトウェアを念頭に置き、デザインとプログラミングのスキルを向上させ、ソフトウェア工学的なセンスを磨くことにある。

フーリエ変換の詳細は各プロジェクト（各員）で調査してもらうが、ここで簡単な概略を記す。実社会には様々な信号（波）がある。株価変動、心電図、音波、オシロスコープなど、いたるところで波のように上下にうねる信号を目にするに違いない。

株価は時が経つに連れて上がったり下がったりする。心電図のようにドックンドックンと周期的に繰り返すものもある。音波は様々な振動数を持つ波が合わさっている。電流や電圧の時間的な変化を波形として観察できるオシロスコープもある。したがって、縦軸は、価格であったり、電位であったり、音の強さであったり、電流であったり、多種多様となるが、横軸はどれも決まって時間であり、上記のいずれの図も情報の流れ（すなわち、信号）を表している。

この手の情報の流れを調査する際の定石は、まず周波数成分に分けて（周波数分析をして）みることである。周波数（frequency）とは1秒間に振動する（繰り返す波の）回数であり、Hz（ヘルツ）という単位をつけて表す。たとえば、次のような1,000個の数値を線でつないだ元データ（1秒間に1,000回のサンプリングを行った離散的なデータ）があったとする。

元データ（1,000個の数値）

これにフーリエ変換（正確には高速な離散フーリエ変換）を施してみると、実部に2つのピーク、虚部にも2つのピークが現れる。これは余弦波（cos波）と正弦波（sin波）が2つずつ、計4つあるということである。

実部（余弦波の周波数が出る） 虚部（正弦波の周波数が出る）

横軸の目盛を数えるとわかるが、12Hzと24Hzの余弦波、5Hzと10Hzの正弦波、これらが成分であることを表す。縦軸は振幅（波の強さ）に関係しており、12Hzの余弦波の振幅は6、24Hzの余弦波の振幅は3、そして、5Hzの正弦波の振幅は4、10Hzの正弦波の振幅は2となっている。これら4つを別々に描き出し、それぞれに対してフーリエ変換を施した結果を並べてみる。

振幅6の12Hzの余弦波 実部（余弦波の周波数が出る） 虚部（正弦波の周波数が出る）
振幅3の24Hzの余弦波 実部（余弦波の周波数が出る） 虚部（正弦波の周波数が出る）
振幅4の5Hzの正弦波 実部（余弦波の周波数が出る） 虚部（正弦波の周波数が出る）
振幅2の10Hzの正弦波 実部（余弦波の周波数が出る） 虚部（正弦波の周波数が出る）

つまり、先ほどの1,000個の元データは上記の4つの波の重ね合わせた波（合成波）だった。それぞれを合計（合成）すると、元データになることを次の図で確認してほしい。それこそがフーリエ変換と逆フーリエ変換を相互に行き来していることに他ならない。

逆フーリエ変換して復元したデータ（1,000個の数値） 結局、4つの波の重ね合わせ

これはちょうど和音の中に含まれるいくつかの単音たちを見つけ出すのに似ている。それがフーリエ変換の順変換であり、その反対が逆変換となる。このようにフーリエ変換をかいつまんでもらっても構わない。実は私たちの聴覚系（脳内）にはフーリエ変換（信号源の分離と合成）がすでに組み込まれていると言っても過言ではない。周波数成分に分解して調査するという分析（解析）法、そして、信号処理の基本（理工学する方法）を垣間見てもらえたと思う。

余談になるが、フーリエ変換を編み出したジョゼフ・フーリエさん（1768年〜1830年）はフランスの数学者である。数学だけに留まらず、熱伝導なども詳細に研究し、物理化学分野の功績も大きい。ナポレオン・ボナパルトさん（1769年〜1821年）とほぼ同世代で、グルノーブルの知事やエジプトの総督も勤めていたので、行政的な手腕もあったのかもしれない。

かつて「白い恋人たち」という曲があった。グルノーブル冬季オリンピック（1968年）のテーマソングである。サザンの桑田佳祐さんの歌にも同じ題名の曲があるし、北海道みやげのお菓子にも単数形の名前を持つものがある。いずれもグルノーブル（の白い恋人たち:skiers）を慕うのであろう。グルノーブル大学の数学教室はフーリエ研究所とも呼ばれている。

フーリエさんの業績は、どんなに込み入った複雑な現象も簡単な現象の寄せ集め（重ね合わせ）であること、複雑な周期振動も難解な関数も単純な正規運動や簡素な関数が重畳（ちょうじょう）したものに過ぎないこと、どんな関数でも詮ずる所は正弦波と余弦波の足し算や引き算などに展開できること、それらを導き出して世に明解に示した（私たちの脳内にあったものを外在化させた）ことである。それをこの演習で追認（追体験）してみようではないか。

フーリエ変換は画像処理や各種データの特性解析などに広く応用できる。ノイズ除去、シグナル強調、特徴検出、領域分割、類似検索、再帰的な周波数分離、肌理（きめ）の解析、欠損情報の復元など、様々な応用に考えを巡らすことも開発の目的である。このプロジェクト演習では、その一環として画像処理に焦点を当てる。画素（ピクセル）の二次元配列である画像を、二次元空間に広がった周期信号とみなし、空間周波数に分解して再構成するアプリケーションの開発である。

基本方針

開発に関わるドキュメントおよびプログラムのすべてをHTML文書にしてWebブラウザからアクセス可能にすること。また、アプリケーションのインストールや取り扱いを説明したマニュアルもHTML文書として作成すること。すなわち、この文書からたどれる全文書（このページの最上部にリンクされた全ページ）を完成させなければならない。以下にそれぞれの基本方針を記載する。

  • 設計方針

    アプリケーションの起動時に一枚のウィンドウを開き、後述する各々の機能をそのウィンドウの中に実現すること。オブジェクト指向設計を行い、UML（Unified Modeling Language）を用いて基本設計書（外部設計書）および詳細設計書（内部設計書）にまとめること。また、ウィンドウアプリケーションを構築する際の定石となっているデザインパターンのObserverパターン：MVC（Model-View-Controller）を採用しなければならない。

  • 管理方針

    開発に伴う仕様書・設計書・報告書などの全ドキュメントおよび全ソースプログラムをリポジトリ（SCM (Software Configuration Management)：SubversionやGitなどのソフトウェア構成管理）の中に格納し、リビジョン管理（バージョン管理）を行い、開発チームで共有すること。そして、このリポジトリ自体を成果物（納品物）に含めなければならない。

  • 実装方針

    コーディング規約を遵守すること。クラスの名称にはアッパーキャメルケース、メソッドの名称にはロワーキャメルケース、リテラルの名称にはアッパーケースやアッパーキャメルケースを用いること。また、変数やメソッドなどの名称はロワーケースやロワーキャメルケースを基本とする。すべての名称は省略せずに意味を明確に示すものを採用すること。リテラル（定数）などは１つのクラス（もしくは名前空間）にまとめ、ソースプログラムの中に散在するのを避けなければならない。

  • テスト方針

    ユーザの操作に関わるテストケースは設計段階でテスト仕様書にまとめること。xUnit（JUnitやSUnitなど）によるテストケースを行うこと。テスト結果は報告書にまとめ、各々のリリースや大きな修正の直後に全件実施（回帰テスト）を行わなければならない。

  • リリース方針

    オープンソースであること。リポジトリからチェックアウトすることにより、全ソースプログラムを獲得できること。また、ソースプログラムから実行モジュールであるアプリケーションを構築するための手段（MakefileやAntのようなコンパイル・リンク・インストール・テスト・アーカイブ作成などを自動化する手立て）を提供すること。そして、アプリケーションは簡単なコマンドによって起動されなければならない。取り扱い説明書（マニュアル）を必ずリリースに含めなければならない。

基本要件

以下に開発計画書および基本設計書を作成するにあって必要なことを記載する。

  • アプリケーションの用途

    まず、画像データを処理する用途に使える。ここで開発するフーリエ変換は信号処理の一種であるから、画像データ処理だけに留まらず、様々なデータの特性解析の用途がある。そして、フーリエ変換を一から作り上げて、オブジェクト指向でデザインし、実装ライブラリとして保持することは宝物となる。研究目的や教育目的で使われるMATLABやMathematicaなどへの過度な依存から脱却する用途も兼ね備える。（バタフライ演算による高速フーリエ変換ぐらい、自力でプログラミングできないようでは、かなりまずい。）

  • 対象となるユーザ

    ノイズ除去、シグナル強調、特徴検出、領域分割、類似検索、再帰的な周波数分離、肌理（きめ）の解析、欠損情報の復元などを行おうとするユーザが対象となる。また、フーリエ変換をMATLABやMathematicaなどに頼った際のロイヤリティ料やライセンス料に悩まされているユーザも対象になるであろう。（研究成果をビジネスにしようとしたり、ベンチャー企業として立ち上げようとする際、当該のロイヤリティ料やライセンス料は製品化や廉価化の大きな障害となっている。）

  • ハードウェア構成

    Name: Apple Macintosh シリーズ　(MacBook / MacBook Air / MacBook Pro / Mac mini / iMac / Mac Pro など)
    Processor: Intel Core 2 Duo 2GHz クラス以上（64bit アーキテクチャの Xenon などを含む）
    Memory: 1GB 以上
    Hard Disk: 250MB 以上の空き容量

  • ソフトウェア構成

    OS: OS X Yosemite (10.10) / OS X El Capitan (10.11) / macOS Sierra (10.12) 以上
    開発に用いるプログラミング言語は以下のいずれかとする。
    1) Java: JDK 1.8 以上
    2) Python: Python 2.7 以上
    3) Smalltalk: VisualWorks 7.6 以上

    加えて、バタフライ演算（J.W.Cooley, J.W.Tukey: An Algorithm for the Machine Calculation of Complex Fourier Series, Mathematics of Computation, Vol.19 1965.）などを導入して必ず高速化を図ること。高速化できなければ、使い物にならない。ソフトウェア工学的に計算量を見積もれば、すなわち、ランダウの記号（O-記法）を用いてアルゴリズムの計算量解析をしてみれば、すぐに判明する。

    一次元の高速な離散フーリエ変換ができあがったら、それを二次元へと拡張し、グレースケール画像やカラー画像に対して適用できるようにすること。横方向（x軸）に一次元のフーリエ変換を施した結果に対して、もう一度、縦方向（y軸）に施す。すなわち、二次元のフーリエ変換は一次元のフーリエ変換の乗算である。

    フーリエさんのグレースケール画像：（128×128・256×256）

    →【フーリエ変換】→ →【逆フーリエ変換】→
    グレースケール画像 パワースペクトル 復元画像

    フーリエさんのカラー画像：（128×128・256×256・256×512・256×293・512×512）

    →【フーリエ変換】→ →【逆フーリエ変換】→
    カラー画像 パワースペクトル 復元画像

    そして最後には、信号処理アプリケーションに仕上げなければならない。一次元においても、二次元においても、フーリエ変換の結果であるパワースペクトルを対話的に編集することで、逆フーリエ変換を施した復元データ（復元画像）を調整できる機能を必ず組み込むこと。その様子を以下に示す。右下のサブウィンドウの中で、右上のパワースペクトルのどこの部分を採用するのかを指定し、それによって左下の復元データが左右されるのを見て取れるであろう。

    以上、ここに記したことを忠実に実現（MVCのウィンドウの中に処理の経過および結果をきちんと可視化）しなければならない。フーリエ変換し復元する一連の処理速度は、画像データ（元データ）の大きさに依存するが、ピクセル数が512×512ぐらいのカラー画像（25万6千個の3倍のデータ）であっても、ストレス無く使用できる（マウス操作でパワースペクトルを編集した際に復元画像が追随して計算される）アプリケーションに仕立て上げること。

  • 制限事項

    Java・Python・Smalltalkの標準ライブラリ（必要であれば拡張ライブラリ）を使用（援用）して開発すること。Macintoshシリーズをターゲットにしているが、実装に用いるJava・Python・Smalltalkはプラットフォーム非依存性を仮想マシンによって実現しているプログラミング言語である。したがって、開発したアプリケーションはLinuxやWindowsなどでも動作することが望ましい。

開発体制

開発に関わるメンバの一覧を作成し、プロジェクトマネージャ、デザイナ、プログラマ、テスタなどの役割を明確化し、加えて、開発するコンポーネント（部品＝クラス）毎に担当者を決めて、それらを開発計画書の中に明記すること。また、仕様適合検査の責任者を立て、次項の開発スケジュールに示されたマイルストーン（目標達成へ向かう過程の一つの段階）ごとに、設計段階から仕様のチェックをクライアント（開発依頼者＝青木淳）と共に行うこと。開発にあたって不明なことが生じた場合には、必ずクライアントと協議して事を進めなければならない。

開発スケジュール

開発スケジュールを前項の開発体制と共に開発計画書の中に明記し、着手する前にクライアントの認可を受ける（契約を取り交わす）こと。実際の開発はプロジェクト演習の15週間で行うこと。授業（15週間分）の30コマ（1コマは90分）の時間を開発に充当させてよい。開発時間が足りない場合には、適宜、他の空き時間を使って、15週間以内での開発（納品＆リリース）を厳守しなければならない。
